01

Le paradigme

Home Assistant (Raspberry Pi) est l'unique système de vérité : capteurs, appareils, scènes, historique. Le dashboard est un SPA React sans backend propre. L'état des entités descend HA → UI par push WebSocket ; l'intention utilisateur remonte UI → HA par appels de services/scènes.

02

Invariants & règles

Les décisions durables qu'un futur constructeur ne peut pas lire dans le code conforme. Chaque bloc : ce qu'il lie, la divergence qu'il empêche, la règle à suivre.

AD-1

HA = unique source de vérité

adopté
Lie
toute donnée partagée / persistée
Empêche
états divergents · backend fantôme
Règle
tout état partagé vit dans HA ; l'app ne possède que de l'état UI éphémère. Aucun backend ni base en v1.
AD-2

Connectivité HA via @hakit uniquement

adopté
Lie
toute I/O vers HA
Empêche
des unités qui ré-implémentent auth/connexion
Règle
accès HA uniquement via @hakit. Exception unique : le média caméra passe par un seul helper.
AD-3

Autorité d'état : une source réactive

Lie
l'état des entités dans l'UI
Empêche
deux widgets aux valeurs obsolètes divergentes
Règle
état confirmé via l'abonnement @hakit seul ; aucun cache persistant. Carve-out : la couche pending d'AD-11.
AD-4

Zéro automatisation côté client (v1)

Lie
où vit le comportement
Empêche
de la logique métier qui fuit dans le client
Règle
l'app appelle services/scènes HA ; toute logique si/alors vit dans HA.
AD-5

Pilotage : optimiste + convergence

Lie
chaque widget de contrôle
Empêche
réconciliation binaire cassée sur volet/clim
Règle
retour visuel <200 ms via AD-11, puis convergence vers la cible ; états transitionnels ≠ échec ; modèle par domaine + timeout.
AD-6

Dégradation à la déconnexion

Lie
tous les widgets (NFR4)
Empêche
écran blanc · entité cloud tombée inaperçue
Règle
obsolescence par entité, indépendante du socket (perte WS ou unavailable) → dernière valeur + indicateur.
AD-7

Mapping entity_id : source unique

Lie
organisation par pièce + widgets
Empêche
ids codés en dur · un concept en 2 entités
Règle
une config unique mappe entity_id ↔ pièce/widget ; une entité canonique par concept (domaine + service).
AD-8

Auth par jeton local

adopté
Lie
authentification (NFR3)
Empêche
ré-auth interactive · token dans le bundle
Règle
token en config gitignorée au runtime, jamais bundlée ; préférer la session HA si servi en même origine.
AD-9

Build statique, servi depuis HA

adopté
Lie
déploiement · NFR5 · NFR1
Empêche
serveur applicatif · démarrage froid lent
Règle
statique (Vite) servi depuis HA, même origine, LAN. PWA : cache app-shell ; données toujours live.
AD-10

Accueil composé + pages profondes

Lie
navigation (FR8, FR9)
Empêche
nav profonde qui casse le glanceable
Règle
un écran d'accueil composé + routes séparées pour l'approfondissement (caméras : live + historique).
AD-11

Couche « pending » par entité

Lie
toute action de pilotage + scène
Empêche
2 widgets sur light.salon qui font la course
Règle
intentions en vol dans une seule couche pending, clé entity_id, last-command-wins, bornée — distincte d'un cache (AD-3).

05

Roadmap & différé

v1 socle domotique

Client HA, intégrations natives, faible risque.

  • Capteurs Netatmo · 4 pièces
  • Éclairage · volets · climatisation
  • Scènes mixtes (dans HA)
  • Caméras Arlo — armer + page live

v2 coordination

Persisté d'abord dans les primitives HA natives.

  • Couche familiale à état partagé
  • Courses · plantes · tortues · corvées
  • UI d'édition des règles (écrit dans HA)
  • Voix : Google → HA → état partagé

Différé au besoin

  • Spike Arlo — mis de côté ; FR8 reste à risque (repli snapshots)
  • Recettes / meal-planning — app dédiée séparée ; le dashboard n'en montre qu'une vue read-only isolée
  • Backlog flux : météo, calendrier, poubelles, conso, colis, trafic
  • Durcissement sécurité · tests / observabilité
