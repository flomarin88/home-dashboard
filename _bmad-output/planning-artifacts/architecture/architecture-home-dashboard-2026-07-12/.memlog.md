- (constraint) PRD-inherited: Home Assistant (Raspberry Pi) is the domotics hub; the dashboard is a FRONT-END over HA, not a from-scratch multi-API integrator
- (constraint) PRD-inherited: target = iPad, always-on kiosk, glanceable home screen + a few deep pages (cameras); touch control
- (constraint) PRD-inherited NFRs: warm home screen interactive <1s + control feedback <200ms; usable by a 5yo (large targets, shallow nav); always-on no re-auth; graceful degradation w/ stale indicator if HA/net down; local-network operation (Netatmo/Arlo are cloud via HA)
- (constraint) PRD-inherited automation split: v1 automations live in HA (dashboard visualizes state + manual override); v2 = dashboard UI to edit rules. Scenes defined in HA, dashboard triggers them
- (constraint) PRD-inherited v1 scope: FR1 Netatmo temp/CO2/humidity 4 rooms; FR2-3 lights; FR4 shutters; FR5 mixed scenes; FR6 AC; FR7 arm/disarm; FR8 Arlo live+history (AT-RISK, snapshot fallback); FR9 kiosk home screen
- (direction) Backend question resolved: HA is the single source of truth / system of record. v1 = NO custom backend, NO DB — React client talks directly to HA over WebSocket. Static build can be served from HA itself.
- (direction) v2 coordination state persists in HA-native primitives first (To-do list entities + input_boolean/input_datetime helpers); voice updates them via HA. Still no custom backend/DB. Dashboard reflects via WebSocket.
- (question) DEFERRED: a dedicated backend+DB (SQLite/Supabase) is introduced ONLY if v2 grows beyond simple shared state (structured recipes/Jow, meal planning, chore history/rewards). Not decided now.
- (decision) PARADIGM (load-bearing): custom React SPA over Home Assistant WebSocket API, using @hakit/core + @hakit/components for HA connectivity (auth, realtime entity state, service calls). HA = sole backend. Binds: every unit's data access + rendering model. Prevents: units re-implementing HA connectivity or diverging on state source.
- (version) @hakit/core & @hakit/components = 6.0.2 (verified npm, published ~June 2026, actively maintained). React major version = seed (owned by code).
- (decision) Build tooling = Vite + React (static build). Binds: no app server. Prevents: a Node/SSR server sneaking in and contradicting the no-backend invariant. TailAdmin React/Vite variant usable for UI.
- (decision) Hosting = static build served same-origin as HA (HA add-on / www folder, or a static host on the same Raspberry Pi). No application server. Binds: deployment topology; Prevents: cross-origin/auth drift and an unplanned backend tier.
- (decision) Auth = single long-lived access token stored on the iPad, used by @hakit. Binds: how the client authenticates; Prevents: interactive re-login (violates NFR3 always-on).
- (decision) Kiosk = PWA (add-to-home-screen, fullscreen) + iOS Guided Access lockdown. Fallback: HA Kiosk app. Binds: the always-on delivery mechanism (NFR3).
- (decision) STATE AUTHORITY: HA entity state is consumed exclusively via @hakit's realtime subscription; no component keeps a separate/duplicate store of HA entity state. Binds: state source; Prevents: two widgets showing divergent stale values.
- (decision) NO CLIENT-SIDE AUTOMATION (v1): the app only calls HA services/scenes; scheduling/if-then logic lives in HA. Binds: where behavior lives; Prevents: business logic leaking into the client. (v2 rule-editing UI writes rules to HA, still does not execute them.)
- (decision) CONTROL ACTIONS = optimistic UI + reconcile on HA state echo (feedback <200ms per NFR1). Binds: how every control widget handles a tap; Prevents: inconsistent laggy vs optimistic behavior across widgets.
- (decision) DISCONNECT DEGRADATION: on WebSocket/HA loss, widgets render last-known value + a staleness indicator, never blank (NFR4). Binds: uniform disconnect handling; Prevents: each widget coping differently / blank screen.
- (decision) ENTITY MAPPING single source: one dashboard config maps HA entity_ids -> rooms (Salon, Chambre Parents, Nathan, Gaspard) and widgets. entity_ids are THE integration contract, referenced in one place, not hardcoded across components. Binds: room/entity ownership; Prevents: components hardcoding entity_ids inconsistently.
- (decision) SCREEN MODEL: one composed glanceable home screen (room/feature widgets + primary controls) + separate routes for deep pages (cameras live+history). Binds: navigation shape.
- (question) DEFERRED (operational): testing/observability strategy — minimal for a hobby kiosk; revisit if reliability becomes a pain.
- (question) DEFERRED (security): long-lived token is a device-local secret on a LAN-only single-home kiosk; hardening (token scope/rotation, TLS on LAN) deferred unless exposure changes.
